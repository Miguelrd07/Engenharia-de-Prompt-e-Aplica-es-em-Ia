# 📌 Projeto Módulo 3 – Low Code / No Code / Vibecode

## 📌 Desafio Escolhido

O desafio proposto pelo grupo foi o desenvolvimento de uma aplicação para cadastro de pessoas, permitindo registrar nome e e-mail de forma simples e organizada.

O objetivo do projeto foi demonstrar, na prática, como ferramentas low code/no code possibilitam a criação de aplicações funcionais sem a necessidade de programação tradicional, atendendo a uma demanda básica de controle e organização de dados.

---

## 🖥️ Protótipo

- O protótipo consiste em uma aplicação com:
  - Tela de cadastro de pessoas (nome e e-mail)
  - Lista com visualização dos registros cadastrados
  - Atualização automática dos dados inseridos

- Funcionamento do protótipo:
  1. O usuário insere o nome e o e-mail no formulário.
  2. Os dados são salvos automaticamente na base da aplicação.
  3. As pessoas cadastradas podem ser visualizadas imediatamente em uma lista organizada.



---

## ⚙️ Plataforma Utilizada

- **Nome da plataforma:** Glide

- **Justificativa da escolha:**  
A plataforma Glide foi escolhida por permitir a criação rápida de aplicativos funcionais a partir de bases simples de dados, além de possuir uma interface intuitiva. Isso facilitou o desenvolvimento do protótipo e possibilitou que a aplicação estivesse totalmente funcional, mesmo utilizando recursos do plano gratuito.

---

## ✅ Vantagens Identificadas

Durante o desenvolvimento do projeto, foram identificadas as seguintes vantagens da abordagem low code/no code/vibecode:

1. Prototipação rápida, com criação da aplicação em pouco tempo.
2. Facilidade de uso, permitindo o desenvolvimento sem conhecimento avançado em programação.
3. Aplicação funcional, com cadastro e visualização de dados funcionando corretamente.

---

## ⚠️ Limitações Encontradas

Apesar das vantagens, algumas limitações também foram percebidas:

1. Customização limitada de layout, restrita aos componentes oferecidos pela plataforma.
2. Dependência da ferramenta, já que a aplicação fica vinculada ao ambiente do Glide.
3. Limitações do plano gratuito, como restrições de recursos mais avançados.

---

## 📚 Reflexão Crítica

O grupo lidou com as limitações focando em uma solução objetiva e funcional, priorizando o atendimento ao desafio proposto. Para contornar as restrições de customização, foram utilizados os componentes disponíveis de forma estratégica, adaptando o fluxo da aplicação às possibilidades da plataforma.

O grupo concluiu que, para projetos maiores ou mais complexos, seria necessário avaliar o uso de planos pagos ou a migração para soluções com maior flexibilidade.
